# 오더 인테이크 프로토콜 (PROTOCOL)

> 사용자(사장/커리어랩장)의 단일 오더가 들어오면 **메인 스레드(=운영체계)** 가 이 절차를 따른다.
> 목적: 하나의 오더 → 영향받는 부서만 자동 연결 → 협업·실행·교차검수·정합·보고까지.
> 트리거: **하이브리드 트리아지**. 모든 규칙은 `CHARTER.md`, 라우팅은 `IMPACT_MAP.md`를 따른다.
> ⚠️ **오더 SoT = Supabase `org_orders`**(웹 `/admin/orders` ↔ 채팅 공용, `admin-tool/scripts/orders-cli.mts`). 아래 절차의 "LEDGER 기록"은 모두 `npm run orders -- …`로 수행한다. `org/LEDGER.md`는 동결 히스토리.

---

## 0. 트리거 — 하이브리드 트리아지

모든 오더는 먼저 **1줄 분류**를 거친다(가벼움, 토큰 최소).

- **직접 처리** (프로토콜 생략): 단순 질문, 단일 파일 조회, 사소한 수정, 단일 부서 명백 작업.
- **풀 프로토콜** (아래 1~10 발동): 다음 중 하나라도 해당 시 —
  - 2개 이상 부서/모듈에 영향 (`IMPACT_MAP.md` §2 참조)
  - 데이터 계약·문서 정합 변경 (§3·§4)
  - 리스크 도메인(`CHARTER.md` §5) 또는 고위험(§4)
  - 판단형 오더("전략", "개선", "왜 안 되지", 성과 판단 등)

> 트리아지 결과를 한 줄로 명시: `[트리아지] 직접처리 / 풀프로토콜 — 사유`.

## 1. 오더 해석
목적(무엇을 달성)·산출물 유형(코드/문서/분석/전략)·성공 기준을 명확화. 모호하면 1회 확인.

## 2. 영향범위 판정 (Impact Map)
`IMPACT_MAP.md`로 **영향 모듈 · 갱신 문서 · 데이터 계약 · 참여 부서 · 리스크 플래그**를 도출.
→ FROZEN 경로 포함 시 **즉시 중단·사장 승인**.

## 3. 관련 부서 자동 선별
Impact Map이 지목한 부서만 spawn. **무관 부서 미참여**(비용·소음 차단).

## 4. 부서별 검토·작업 분해
각 본부장: 자기 영역 영향 점검 → 필요 변경·산출물·교차 의존을 반환. **근거 인용 필수**(`CHARTER.md` §1).

## 5. 협업 요청 생성
본부 A의 변경이 본부 B 영역을 침범하면 → `LEDGER.md`에 협업요청 기록 → 메인이 B를 spawn. (부서 간 소통은 메인 스레드가 중개)

## 6. 실행 (자율성 경계)
- 저위험 → 실행 에이전트(코드 6종) 위임.
- 고위험·리스크 도메인 → `LEDGER.md` `승인대기` → 사장 표면화. 승인 전 실행 금지.

## 7. 교차 검수 (실행 ≠ 검수)
`CHARTER.md` §6 페어링으로 **다른 부서**가 검수. 리스크 도메인은 IT·보안실 추가 검수.

## 8. 문서·데이터·코드 정합
`IMPACT_MAP.md` §4 정합 체인 수행 → `/sync-arch`로 검증. **미갱신 문서 0**.

## 9. 최종 보고 (두괄식) → 커리어랩장/사장
1문장 결론 → 3문장 요약 → 세부(참여부서·변경·검수결과·근거·잔여리스크).
`LEDGER.md` 상태 갱신(완료/검증/승인대기).

## 10. 피드백 루프
사장/커리어랩장 피드백 → `LEDGER.md`에 신규 지시 → **1단계 재진입**. 미반영 피드백 0 유지.
